4. Authentication — How Login Works
Technology: JWT (JSON Web Token)
What is JWT?
When you log in, the server gives you two tokens:

Access Token — proves who you are (expires in 60 minutes)
Refresh Token — used to get a new access token (expires in 7 days)

Flow:
User types email + password
        ↓
POST /api/auth/login/
        ↓
Django checks credentials
        ↓
Returns: { access_token, refresh_token, user }
        ↓
React saves tokens to localStorage
        ↓
Every future API request sends:
  Authorization: Bearer <access_token>
Token Refresh (auto-renewal):
API call returns 401 (token expired)
        ↓
api.js interceptor catches it
        ↓
POST /api/auth/token/refresh/
        ↓
Get new access token silently
        ↓
Retry the original request
        ↓
User never notices anything
Files involved:

accounts/views.py → CustomTokenObtainPairView
frontend/utils/api.js → interceptor handles refresh
frontend/context/AuthContext.jsx → stores user state

## 8. WebSockets — How Real-Time Messaging Works

**Technology:** Django Channels

### Why WebSockets?
Normal HTTP is **request-response** — the browser has to ask the server for new messages. WebSockets keep a **persistent connection** open so the server can push messages instantly.
```
Normal HTTP:
  Browser → "Any new messages?" → Server → "No"   (every few seconds)

WebSocket:
  Browser ←——————— open connection ———————→ Server
                    Server pushes instantly when message arrives
```

### How Django Channels Works:
```
HTTP requests  → Django views (normal)
WebSocket      → Django Channels Consumers (async)
ASGI (Asynchronous Server Gateway Interface) handles both:
python# asgi.py
ProtocolTypeRouter({
    'http':      → Django views
    'websocket': → Channels consumers
})
```

### WebSocket Authentication Problem:
Normal HTTP sends cookies/headers automatically. WebSockets don't support the `Authorization` header easily.

**Solution:** Pass the JWT token in the URL:
```
wss://localhost:8443/ws/chat/4/?token=eyJhbG...
```

The `JWTAuthMiddleware` reads this token and sets `scope['user']` — similar to how `request.user` works in normal Django views.

### Two WebSocket Connections:

**1. Chat Socket** (`/ws/chat/<room_id>/`)
- One per open chat room
- Handles: typing indicators, read receipts
- Receives new messages broadcast from the REST API

**2. Signal Socket** (`/ws/signal/`)
- One per logged-in user, always open
- Handles: online/offline presence, WebRTC call signals

### Message Flow (sending a message):
```
User types message → clicks Send
        ↓
React: POST /api/chat/rooms/4/messages/send/
        ↓
Django SaveMessageView saves to database
        ↓
Django calls channel_layer.group_send('chat_4', message)
        ↓
ChatConsumer.chat_message() runs for ALL connected clients
        ↓
Both users see the message instantly
```

## 9. WebRTC — How Voice/Video Calls Work

**Technology:** WebRTC (Web Real-Time Communication)

### Key concept: Peer-to-Peer
Unlike messages (which go through the server), WebRTC calls go **directly browser-to-browser**. The server is only used to help the two browsers **find each other** — this is called **signalling**.

### Call Flow:
```
User A clicks "Call"          User B
      │                           │
      │── call_offer (SDP) ──────►│   (via Signal WebSocket)
      │                           │ User B clicks Accept
      │◄─ call_answer (SDP) ──────│   (via Signal WebSocket)
      │                           │
      │◄──── ICE candidates ──────│   (helps find network path)
      │───── ICE candidates ─────►│
      │                           │
      └─────── Direct P2P audio/video stream ──────┘
                   (no server in the middle)
What is SDP?
SDP (Session Description Protocol) — a text description of:

What codecs the browser supports (H.264, VP8, Opus)
What media tracks are offered (audio only / audio+video)

What are ICE Candidates?
ICE candidates are possible network paths between the two browsers (local IP, public IP, via STUN server). Both browsers share all their candidates and try each one until they find a path that works.
STUN Server:
pythonICE_SERVERS = {
    'iceServers': [
        { 'urls': 'stun:stun.l.google.com:19302' }
    ]
}
```
STUN (Session Traversal Utilities for NAT) helps each browser discover its **public IP address** so it can share it with the other browser. We use Google's free STUN server.

## 10. Frontend Architecture

### State Management:
No Redux. Uses **React Context + hooks** because the app is simple enough.
```
AuthContext (global)
  └── user, login(), logout(), register()
      Available to every component via useAuth()

Chat.jsx (local state)
  └── rooms, messages, room, call, isRecording...
```

### Custom Hooks:

**`useChatSocket(roomId, onMessage)`**
- Opens a WebSocket to `/ws/chat/<roomId>/`
- Reconnects automatically if connection drops
- Closes old connection when user switches rooms
- Returns `{ send }` function

**`useSignalSocket(onMessage)`**
- Opens one persistent WebSocket to `/ws/signal/`
- Always connected while logged in
- Returns `{ send }` function

11. SSL / HTTPS Setup
Why HTTPS locally?

WebRTC getUserMedia() (microphone/camera) only works over HTTPS
Secure WebSockets (wss://) require HTTPS
CORS with credentials requires HTTPS

How:

generate_ssl.py creates cert.pem + key.pem (self-signed)
Daphne (ASGI server) serves everything over SSL:

bashdaphne -e ssl:8443:privateKey=key.pem:certKey=cert.pem -b 0.0.0.0 chatproject.asgi:application

Browser shows "Not Secure" warning → click Advanced → Proceed (accepts self-signed cert)

13. How to Explain to Your TL in 2 Minutes

"This is a full-stack real-time chat app. The frontend is React, backend is Django. Users log in with email and JWT tokens — the token is sent with every API request so Django knows who's calling.


For real-time messaging, we use WebSockets through Django Channels. When you send a message, it hits the REST API, gets saved to the database, then the server broadcasts it through the WebSocket so the other person sees it instantly.


For voice and video calls, we use WebRTC. The two browsers connect directly to each other — the server only helps them find each other using a STUN server and our Signal WebSocket. Once connected, audio and video stream peer-to-peer.


We run everything through Daphne with a self-signed SSL certificate because WebRTC requires HTTPS to access the microphone and camera."
